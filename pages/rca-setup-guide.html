1 Enable Revenue Cloud Features

This step has three sub-steps that must be done in order. The "Enable Revenue Cloud Features" toggle on the Revenue Settings page will be greyed out until the prerequisites below are completed.

1a. Enable New Order Save Behavior (Release Update)

Setup → Release Updates → "Enable New Order Save Behavior"

Click "Enable Test Run" on this release update. This is a hidden prerequisite — the Revenue Cloud Features toggle will remain greyed out without it.

After the New Order Save Behavior is enabled, whenever an order product update causes a change to the parent order, Salesforce runs custom application logic (validation rules, Apex triggers and classes, workflow rules, and flows). Previously, Salesforce did not correctly evaluate custom application logic on the parent record in this scenario. — Based on Haley Bates' RCA training documentation

Package Compatibility Warning: Packages with orders created before Winter '21 don't support the New Order Save Behavior release update by default. If you use a package involving orders, wait for confirmation from your package provider before enabling the release update.
Undocumented Prerequisite: Salesforce does not document this dependency. The "Enable New Order Save Behavior" release update must be activated (via "Enable Test Run") before the Revenue Cloud Features toggle becomes clickable. This is confirmed by Salesforce Support (Case #472941736, March 2026) — other customers have reported the same issue. "Revenue Lifecycle Management Settings" may not appear in Setup search; use the Revenue Settings page under Feature Settings → Revenue Cloud instead.

1b. Enable Order Settings Prerequisites

Setup → Order Settings

Check all of the following:

  • Enable Orders
  • Enable Reduction Orders
  • Enable Negative Quantity
  • Enable Zero Quantity
  • Enable Enhanced Commerce Orders — Warning: cannot be disabled once enabled

1c. Enable Revenue Cloud Features

Setup → Feature Settings → Revenue Cloud → Revenue Settings

After completing 1a and 1b:

  • "Enable New Order Save Behavior" prerequisite shows Enabled
  • "Enable Revenue Cloud Features" toggled ON
  • Green banner "Revenue Cloud has been enabled" confirmed
  • Salesforce Pricing toggled ON (Step 1d)

Standalone Permission Sets (3) — Assign Individually

Security Warning: Some RCA Permission Sets grant Run Flows or Manage Flows, giving full control over ALL flows in the org. Use Muting Permission Sets within your PSGs to remove these risky permissions for non-admin users.
CLI Bug: sf data query does NOT return Standard-type (PSL-based) permission set assignments. Only Regular-type PS appear in results. Use Apex to query PermissionSetAssignment with WHERE PermissionSet.Type = 'Standard' to see the full picture. The Setup UI always shows all assignments correctly.
PSG Creation Order: Create the PSG first → assign to user (can assign empty) → then add Permission Set components. Adding components to an already-assigned PSG propagates automatically if compatible. The PSG Status field shows "Updating" while recalculating and "Updated" when ready.
Assignment Order: Always assign in this order: PSLs first (Step 2) → PSGs (Step 3) → Standalone PS last. Assigning out of order causes failures.
Debugging New PSG Conflicts: If you encounter a new aggregation conflict when adding a PS to a PSG, use the binary search method: assign an empty PSG to the user, then add components one at a time (or in batches) to isolate which PS causes the conflict. The error message will name the conflicting permission.
Deploying PSGs: PSGs are deployable via change sets (Component Type: Permission Set Group). The individual Standard PS inside them are Salesforce-managed and don’t need deployment — only the PSG container and its component linkages deploy. PSL assignments are NOT deployable and must be re-done in the target org before assigning PSGs.

4e. Extend Context Definitions

Setup → Feature Settings → Context Service → Context Definitions

Context Definitions tell the Pricing Procedure what data fields are available during pricing calculations. Salesforce ships two standard definitions, but you must extend them to create custom versions that can be modified and linked to your procedures. This is required before configuring Pricing Procedures (Step 7).

UI-Only Operation: Extending Context Definitions cannot be done via CLI, API, Tooling API, or Metadata API. This must be done manually in the Setup UI. However, once created, the extended definitions are deployable via change sets or metadata packages.

Why Extend?

The standard (OOTB) context definitions are read-only — you can't modify them, set them as default, or link them to your Pricing Procedures. Extending creates a custom copy that inherits all standard mappings while allowing you to add custom fields, change mappings, and set it as the default for your org.

4e-i. Extend SalesTransactionContext

This context definition maps Quote and QuoteLineItem fields into the pricing engine. It's used by the Revenue Management Default Pricing Procedure.

  • Navigate to Context Definitions → Standard Definitions tab
  • Find SalesTransactionContext and click on it
  • Click the Extend button (top-right area of the definition page)
  • Name it EXT_SalesTransactionContext (Salesforce will auto-prefix with your namespace if applicable)
  • Go to the Map Data tab → Click Edit SObject Mapping → Check "Mark as Default" → Save
  • Click Activate on the extended definition
  • Verify: Status shows Active and it appears on the Custom Definitions tab

4e-ii. Extend ProductDiscoveryContext

This context definition maps product catalog fields into the pricing engine. It's used by the Product Discovery Pricing Procedure (the "Browse Catalog" experience).

  • Find ProductDiscoveryContext on the Standard Definitions tab
  • Click Extend
  • Name it EXT_ProductDiscoveryContext
  • Go to Map Data tab → Edit SObject Mapping → Check "Mark as Default" → Save
  • Click Activate
  • Verify: Status shows Active, appears on Custom Definitions tab
Effective Date Dependency: Note the Effective From date on your extended context definitions. When you create Pricing Procedures in Step 7, the procedure's Start Date Time must be at least one day AFTER this date. Mismatched dates cause silent pricing failures with no error message.
Adding Custom Fields Later: To add custom fields to pricing calculations (e.g., customer tier, region), return to the extended context definition → Map Data tab → Edit → click SalesTransactionItem → create a new attribute (set INPUT/OUTPUT to the same data type) → create a tag → then map it to the corresponding QuoteLineItem field in the SObject Mapping. See Step 12 for grouping field examples.

7 Configure Pricing Procedures

Do NOT build pricing procedures from scratch. Always clone from Salesforce-provided templates. Missing required elements will cause pricing to fail silently.
UI-Only Operation: Pricing Procedures must be created from Expression Set Templates via "Save As" in the UI. This cannot be automated via CLI, API, or metadata deployment. However, once created, pricing procedure metadata can be included in change sets.

Understanding Expression Set Templates

Expression Set Templates are found via App Launcher → Expression Set Templates. These are templates of the different procedures that need to be set up for RCA. When you click to open one, it takes you to a builder that looks like Flow where you create the procedure. — Haley Bates

7a. Clone the Revenue Management Default Pricing Procedure

App Launcher → Expression Set Templates
  • Open the "Revenue Management Default Pricing Procedure" template
  • Click Save As → New Pricing Procedure
  • Set Rank = 1
  • Keep it inactive so you can change the context definition to your custom one first
  • Save the new Pricing Procedure
Important: On the new Pricing Procedure, click the cog wheel (settings) to the right of the page. Ensure that the Start Date Time is set at least one day AFTER the Start Date Time specified on the custom Context Definition. Mismatched dates will cause silent failures.

7b. Link the Custom Context Definition

App Launcher → Pricing Procedures
  • Navigate to App Launcher → Pricing Procedures
  • Change Context Definition to custom SalesTransactionContext
  • Save
  • Open Pricing Procedure version → Activate from the builder

7c. Clone the Product Discovery Pricing Procedure

Repeat the same process for the Product Discovery procedure, but use the custom ProductDiscoveryContext definition instead:

  • Open "Product Discovery Pricing Procedure" template from Expression Set Templates
  • Save As → New Pricing Procedure → Set Rank → Keep inactive
  • Link to custom ProductDiscoveryContext definition
  • Activate the procedure version

7d. Link Pricing Procedures to Settings

After creating and activating both procedures, connect them in three places:

  • Product Discovery Settings — Set Context Definition to custom ProductDiscoveryContext, set Pricing Procedure to Product Discovery procedure
    Setup → Product Catalog Management → Product Discovery → Product Discovery Settings
  • Salesforce Pricing Setup — Set Pricing Procedure to Revenue Management Default
    Setup → Feature Settings → Salesforce Pricing → Salesforce Pricing Setup
  • Revenue Settings — Set Pricing Procedure to Revenue Management Default
    Setup → Feature Settings → Revenue Cloud → Revenue Settings

Pricing Procedure Architecture

Context Definitions

If you need custom fields in pricing calculations:

  1. Create extended versions of standard context definitions (see Step 4e)
  2. Add custom fields to the appropriate section (Sales Transaction or Sales Transaction Item)
  3. Map to the correct Quote or QuoteLineItem fields

Decision Tables

Decision Tables replace CPQ Price Rules. They are structured lookup tables:

  • Conditions: Product, Selling Model, Date range, Customer attributes
  • Outputs: Price, Discount %, Adjustment Amount
  • Support complex multi-dimensional lookups

8 Set Up Product Catalog

8a. Create Product Selling Models

App Launcher → Product Catalog Management
  • One-Time (OneTime) — Single purchases, hardware, professional services
  • Term-Defined Annual (TermDefined, PricingTerm=1, PricingTermUnit=Annual) — Annual subscriptions
  • Evergreen (Evergreen, PricingTerm=1, PricingTermUnit=Months) — Month-to-month services
  • Usage-Based (UsageBased) — Consumption pricing, metered billing
  • All Product Selling Models activated (Status = Active)
PricingTermUnit gotcha: The PricingTermUnit restricted picklist uses Annual (not "Years") and Months. For Term-Defined annual models, set PricingTerm = 1 with PricingTermUnit = Annual. For Evergreen monthly, use PricingTerm = 1 with PricingTermUnit = Months. These are restricted picklist values — query Schema.DescribeFieldResult in your org to confirm available options.

8b. Create Catalog Structure

  • Create a ProductCatalog (e.g., "Enterprise Catalog")
  • Create ProductCategory hierarchy (e.g., Hardware → Software → Services)

8c. Create Products

Set ProductType at creation time. The ProductType field (Simple or Bundle) cannot be changed after creation (Spring ’25+). Decide before creating the record. There is no going back.
  • Create Product2 records with correct ProductType (Simple or Bundle)
  • Set IsActive = true
  • Assign ProductSellingModelOption (junction linking product to selling model)
  • For bundles: use the Structure tab to add components and groups
  • Run "Validate Product Definition" before activating bundles

8d. Create Price Book Entries

  • Verify the Standard Price Book is active (required before creating entries)
  • Add products to Price Book with List Prices
  • Link ProductSellingModel on each PricebookEntry
Tip: One product can have multiple ProductSellingModelOptions, allowing you to sell the same product as one-time OR subscription without creating duplicate SKUs.

8e. Build and Search Index

App Launcher → Product Catalog Management → Build and Search Index

The product search index powers the Browse Catalog experience in quotes. Without it, users cannot search or add products to quotes.

Products Required First: "Create Full Index" will fail with "No products are assigned to any catalog" if no products have been assigned to categories yet. You must complete steps 8b–8d before building the index.
  • Navigate to Build and Search Index tab → Click Create Full Index
  • Verify: Index Status shows Active with product count > 0
  • Review Manage Fields & Attributes tab for searchable/filterable fields (optional)
Rebuild Required: You must rebuild the index whenever you update product information, add new products, change categories, or modify search options. This is a manual step that is easy to forget.

9 Sync Pricing Data & Test

Sync Pricing Data

Setup → Sync Pricing Data → Execute
CRITICAL for Initial Setup: You MUST run "Sync Pricing Data" at least once during initial setup to populate the pricing engine with your product and price book data. Without this, pricing will not work at all.
Ongoing Changes — Use Decision Tables Refresh Instead: After the initial sync, do NOT use "Sync Pricing Data" for every change — it burns through your org’s allotment. For ongoing product/pricing changes (new PBEs, updated prices), use Decision Tables Refresh instead. — Haley Bates
  • Run Sync Pricing Data (initial setup only)

Refresh Decision Tables

Setup → Decision Tables → Price Book Entries → Refresh

The preferred method for ongoing pricing changes. Ensures new pricing data is available for quoting operations without burning your sync allotment.

  • Decision Tables refreshed after product/pricing changes

Test Your Configuration

  • Create a test Quote (Quotes → New)
  • Click Browse Catalog and select products
  • Verify pricing calculates correctly
  • Test bundle configuration (if applicable)
  • Generate a test order from the quote

10 Set Up a Product Bundle

Bundles in RCA use ProductComponentGroup and ProductRelatedComponent instead of CPQ's ProductFeature/ProductOption. Groups are required for configurable bundles.

CRITICAL: ProductType cannot be changed after creation. Decide if a product is "Bundle" or "Simple" before creating it. There is no going back.

10a. Create the Bundle Product

App Launcher → Product Catalog Management → Products → New
  • Set ProductType = Bundle
  • Set IsActive = true
  • Set ConfigurableDuringSale = true (if reps should modify components during quoting)
  • Set IsAssetizable = true (if the bundle creates assets on order activation)
  • Add ProductSellingModels and PricebookEntries for the bundle

10b. Create Component Groups

Product Record → Structure Tab → Add Group
  • Click Add Group and name it (e.g., "Compute Options", "Support Tier")
  • Set Cardinality (min/max selections) — e.g., Min=1/Max=1 for radio-button, Min=0/Max=3 for optional
  • Set Sequence for display order
  • Repeat for each component group

10c. Add Child Products to Groups

Product Record → Structure Tab → Group → Add Component
  • Click Add Component and select child product (must already exist and be Active)
  • Set Default Quantity
  • Set Is Required = true for mandatory components
  • Configure BundleBasedAdjustment for bundle-specific pricing (optional)

10d. Validate and Test

  • Click Validate Product Definition on the Product record
  • Fix any issues flagged
  • Run Sync Pricing Data (if pricing was changed)
  • Test Quote → Browse Catalog → add bundle → verify configurator shows groups with correct cardinality
Static vs. Configurable: If ConfigurableDuringSale = false, the bundle has a fixed configuration and groups are optional. If true (configurable), ProductComponentGroup records are required.

11 Configure the Price Waterfall

The price waterfall defines the sequence of pricing calculations — from base list price to final net price. Each step is a Pricing Element within a Pricing Procedure.

11a. Clone Pricing Procedure Templates

Setup → Expression Set Templates
  • Clone "Revenue Management Default Pricing Procedure" and name descriptively
  • Clone "Product Discovery Pricing Procedure"
  • Set Activation Start/End Dates on each clone
CRITICAL: Never build pricing procedures from scratch. Always clone from templates. Missing required internal elements causes silent pricing failures.

11b. Configure Context Definitions

Setup → Context Definitions
  • Create an EXT (Extended) Context Definition if you need custom fields
  • Map Sales Transaction → Quote fields (Account, Opportunity, custom fields)
  • Map Sales Transaction Item → QuoteLineItem fields (Product, Quantity, etc.)
  • Add any custom fields your pricing logic needs (e.g., customer tier, region)

11c. Build Decision Tables

Setup → Decision Tables
  • Create a Decision Table (e.g., "Volume Discount Entries")
  • Define Input columns (conditions): Product, Quantity Range, Customer Tier
  • Define Output columns (results): Discount Percentage, Adjustment Amount
  • Add rows — each row is one pricing rule

11d. Add Pricing Elements to the Procedure

Open your cloned Pricing Procedure in the Expression Set designer. Add elements in waterfall order:

OrderElementPurpose
1List PriceBase price from PricebookEntry
2Volume / Tier DiscountQuantity-based tiers via PriceAdjustmentSchedule
3Attribute-Based AdjustmentPrice changes from product configuration
4Bundle-Based AdjustmentBundle-specific pricing (vs. standalone)
5Contract-Based PricingAccount-specific negotiated rates
6Formula-Based PricingCustom calculations
7Manual DiscountSales rep adjustments
8Subscription PricingMulti-period recurring calculations
9ProrationPartial-period adjustments
10Price TrackingFloor/ceiling enforcement

11e. Link and Sync

  • Associate Pricing Procedure with transaction type (Setup → Sales Transaction Types)
  • Run Sync Pricing Data
  • Refresh Decision Tables
  • Test using Simulation feature in the Pricing Procedure designer
Tip: Use the Simulation feature's Basic mode to quickly test with a product, or Advanced mode with JSON input for precise testing. The Price Waterfall log shows each element's input, calculation, and output for debugging.

12 Use Quote Line Groups

RCA has a native QuoteLineGroup object (API v61.0+) managed through the Transaction Line Editor (TLE). This replaces CPQ's SBQQ__QuoteLineGroup__c.

Prerequisites

  • Deploy the Sales Transaction Line Editor (LWC) component on Quote and Order Lightning record pages
  • If using ramp deals: enable in Revenue Settings and activate "Clone Quote Line Groups"

Manual Grouping

  1. Open a Quote → navigate to the Transaction Line Editor
  2. Click "Add Group" in the TLE toolbar
  3. Name the group (e.g., "Year 1 Licenses", "Professional Services")
  4. Select line items using checkboxes
  5. Click "Move Selected Lines" in the group action header
  6. Subtotals appear at group level automatically

Automatic Grouping

Automatic grouping is possible based on fields or field values. Manual grouping is also supported. Order line groups are created from quote line groups when an order is created from a quote. — Haley Bates

  1. Click the Group action in the TLE toolbar
  2. Select a column to group by (e.g., Product Family, Category)
  3. Lines auto-organize into groups — names are editable
  4. Use "Ungroup" to return to flat view

Grouping Field Compatibility

Not all field types work for automatic grouping:

Supported Field TypesUnsupported Field Types
TextNumber
PicklistDate
Formula (must use fields on the QuoteLineItem object only — no cross-object fields)Lookup
Cross-object formulas
Limitation: Salesforce Revenue Cloud cannot group by cross-object fields or complex formulas. Only formulas referencing fields directly on the QuoteLineItem object are supported.

Adding Grouping Fields to Context Definition

You must make any grouping fields available within the Pricing Procedure's context definition. — Haley Bates

  1. Find your custom SalesTransactionContext definition in Setup
  2. Go to the Map Data tab
  3. Click Edit at the top right to create a new attribute
  4. Click SalesTransactionItem to create the attribute (set INPUT/OUTPUT to the same data type)
  5. Click Next, then click SalesTransactionItem, find the new attribute and create a tag for it (use the same name as the attribute, no spaces)
  6. Save, then Edit the QuoteEntitiesMapping on the Map Data tab to map the new attribute to the corresponding QuoteLineItem field

Ramp Deals for Groups (Winter '26+)

  1. Create or select a group in the TLE
  2. Mark the group as a "Ramp Group"
  3. Define start/end dates and segment types
  4. Products within the group follow the ramp schedule together
  5. Clone ramp segments year-over-year with uplift logic
Tip: With Winter '26+, you can generate separate orders from groups — each QLG becomes its own order, or you can group by a specific line field value.

Tip: The start and end dates on the invoice line determine the billing period displayed on the invoice PDF. Ensure these are populated correctly for accurate invoice output.

A Automation Script

A setup script is provided at scripts/rca-org-setup.sh that automates steps 2, 3, 5, and creates starter data (Product Selling Models, default Catalog with categories).

Usage

# Authenticate to your org first
sf org login web -a myorg

# Run the setup script
./scripts/rca-org-setup.sh myorg

What the Script Does

  1. Validates org connection
  2. Assigns all RCA Permission Set Licenses to current user
  3. Assigns all RCA Permission Sets to current user
  4. Deploys ConstraintEngineNodeStatus custom fields
  5. Validates RCA object accessibility (runs test Apex)
  6. Creates standard Product Selling Models (One-Time, Term-Defined, Evergreen)
  7. Creates default Product Catalog with sample categories
  8. Outputs remaining manual steps
Note: The script handles errors gracefully — PSLs/PSs that don't exist in your org edition will be skipped. Review the output for any items that need attention.
Known Issue — PSL DeveloperName Mismatch: The rca-org-setup.sh script uses PSL DeveloperNames that may not match your org. For example, the script references RevCloudUserPsl but the actual name is RevenueLifecycleManagementUserPsl, and ProductCatalogManagementAdminPsl should be ProductCatalogManagementAdministratorPsl. If PSL assignments fail, run SELECT DeveloperName FROM PermissionSetLicense WHERE DeveloperName LIKE '%Revenue%' OR DeveloperName LIKE '%Product%' OR DeveloperName LIKE '%Pricing%' to get the correct names for your org.
Known Limitation — Single User Only: The sf CLI can only assign PSLs/Permission Sets to the currently authenticated user. To assign to multiple users, use Anonymous Apex with bulk insert patterns (loop through users and PSLs, skip existing assignments, use try/catch for individual failures).
